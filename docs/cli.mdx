# CLI

`ddb` is a command line interface (CLI) to access DroneDB's functions. This makes it ideal for power users and for creating automated workflows.

## Installation


<Tabs>
  <TabItem value="windows" label="Windows" default>

- Download [the latest release](https://github.com/DroneDB/DroneDB/releases/latest) from GitHub
- Extract the archive and add the folder to your PATH
- Verify it's working:

```bash
C:\> ddb --version
x.x.x (git commit xxxxxxx)
```

:::note
The version number will vary depending on the release you downloaded.
:::

  </TabItem>
  <TabItem value="linux" label="Linux (Debian/Ubuntu)">

Download the `.deb` package from the [latest release](https://github.com/DroneDB/DroneDB/releases/latest) and install it:

```bash
$ sudo dpkg -i dronedb_x.x.x_amd64.deb
$ ddb --version
x.x.x (git commit xxxxxxx)
```

:::note
Only Debian-based distributions (Debian, Ubuntu, Linux Mint, etc.) are officially supported.
:::

  </TabItem>
</Tabs>

## Examples

You can perform all sorts of operations with `ddb`. We recommend checking out
the list of [commands](#commands-reference):

### Glob patterns

`ddb` accepts shell-style glob patterns wherever a path or list of paths is
expected (for example with `add`, `share`, `list`, `info`):

| Pattern | Matches |
| --- | --- |
| `*.JPG` | All `.JPG` files in the current directory |
| `images/*.tif` | All `.tif` files directly under `images/` |
| `**/*.las` | All `.las` files in the current directory and any subdirectory |
| `images/**/*` | All files under `images/`, recursively |
| `images/` | The directory itself plus everything under it (recursive) |

Some shells (POSIX shells, PowerShell) expand globs before passing them to
`ddb`. Windows `cmd.exe` does not — `ddb` performs the expansion for you in
that case. Quote the pattern (e.g. `'**/*.JPG'`) if you want to force `ddb`
to handle the expansion itself.

The `.ddb` index folder is always skipped. Patterns that match nothing print a
warning, and `ddb` exits with an error if no patterns matched anything at all.

### Sharing datasets

From a directory with images, simply type:

```bash
ddb share *.JPG
```

:::tip
By default the images are shared with [Hub](https://hub.dronedb.app). You will need to
[register an account](https://dronedb.app) to get a username and
password. But you can also self-host your own
[Registry](./registry).
:::

You can select a different server by typing:

```bash
ddb share *.JPG -s http://localhost:5000
--> https://localhost:5000/r/admin/193514313aba4949ab5578b28ba1dd5b
```

Because we didn't set a `tag`, ddb generated a random one for you.
You can change the tag by visiting the URL. You can also explicitly set
a tag, like so:

```bash
ddb share *.JPG -t pierotofy/brighton
```

Or if you're running your own Registry:

```bash
ddb share *.JPG -t http://localhost:5000/admin/brighton
```

Tags are defined as:

`[server]/organization/dataset`

With the server component being optional.

### Editing datasets

Using the Web UI provided by Registry is the easiest way to make
changes.

You can also `clone` (download) an existing dataset from a
Registry for offline use, make modifications, then sync back your
changes.

Let's use this dataset:
[https://hub.dronedb.app/r/pierotofy/brighton-beach](https://hub.dronedb.app/r/pierotofy/brighton-beach)

You can clone it via:

```bash
ddb clone pierotofy/brighton-beach
```

Let's add a `README.md` file that describes the dataset. Create a
`README.md` file using
[Markdown](https://www.markdownguide.org/cheat-sheet/) syntax and save
it in the `brighton-beach` directory. Afterwards:

```bash
cd brighton-beach/
ddb add README.md
```

Great! We are now ready to push the changes.

```bash
ddb push
```

Uuups! This will trigger an error, since we don't have permission to
make modifications to this dataset (it belongs to `pierotofy`). Let's
make our own copy to a different Registry server and user:

```bash
ddb tag http://localhost:5000/admin/brighton-copy
ddb push
```

### Metadata Entries

DroneDB supports the addition of metadata to any file or directory
within the index. This can be used to store information of any kind in
JSON format:

```bash
ddb meta set pilot '{"name": "John Smith"}'
ddb meta get pilot --format json
# Output: {"data":{"name":"John S."},"id":"ff4f0f26-8741-4423-bde5-b445750937bb","mtime":1640985850}

ddb add photo.JPG
ddb meta add comments '{"text": "Nice one!", "author": "John S."}' -p photo.JPG
ddb meta get comments -p photo.JPG --format json
# Output: [{"data":{"author":"John S.","text":"Nice one!"},"id":"550d0b5c-108b-4996-b7e8-467b4cb87937","mtime":1640986217}]
```

Singular and plural metadata keys are supported. Plural keys (ending
with `s`) are treated as lists, whereas singular keys are objects.

Metadata entries are synced on push/pull and people working on the same
dataset while offline can later sync back online without conflicts.

### Projecting images

![geoproject](https://user-images.githubusercontent.com/1951843/176517923-d391fc9b-24cb-4604-a134-51be9155c806.png)

You can project images onto a map.

```bash
ddb geoproj images/DJI_0018.JPG -o projected/
```

### Creating static tiles (XYZ/TMS)

DroneDB can create static tiles for GeoTIFFs, drone images and [EPT](https://entwine.io). It's similar to
[gdal2tiles.py](https://gdal.org/programs/gdal2tiles.html), but
it's faster and can handle more formats. You can use
these tiles in applications such as Leaflet or OpenLayers to display
them on the web.

```bash
ddb tile DJI_0018.JPG output_tiles/
ddb tile ept.json output_tiles/
ddb tile orthophoto.tif output_tiles/
```

### Building point cloud tiles (EPT)

DroneDB can build EPT (Entwine Point Tiles) indexes for efficient streaming of point clouds:

```bash
ddb ept output/ point_cloud.laz
```

This creates a hierarchical structure that enables fast visualization of large point clouds in web browsers.

DroneDB supports multiple point cloud formats including LAS, LAZ, E57, PTS, XYZ, and PLY. XYZ files benefit from improved coordinate parsing and validation logic for more reliable indexing.

### Generating Cloud Optimized GeoTIFFs (COG)

Convert standard GeoTIFFs to Cloud Optimized GeoTIFFs for efficient streaming:

```bash
ddb cog output.tif input.tif
```

COGs enable range requests, making it possible to load only the needed portions of large rasters.

### Extracting GPS locations/footprints to GeoJSON

You can quickly extract the locations of images to GeoJSON via:

```bash
ddb info *.JPG -f geojson -o gps.geojson
```

Or for image footprints:

```bash
ddb info *.JPG -f geojson --geometry polygon -o footprint.geojson
```

This works with orthophotos, elevation models and point clouds as well!

```bash
ddb info point_cloud.laz -f geojson --geometry polygon -o footprint.geojson
```

### Generating STAC Catalogs

DroneDB can generate STAC (SpatioTemporal Asset Catalog) compliant JSON for interoperability with other geospatial tools:

```bash
cd my-dataset/
ddb stac
```

For a specific file:

```bash
ddb stac -p orthophoto.tif
```

### Building 3D Model Tiles (Nexus)

For efficient web streaming of 3D meshes, DroneDB can generate Nexus files:

```bash
ddb nxs model.obj output.nxz
```

Supported input formats: OBJ, GLTF, GLB.

### Generating Thumbnails

Create thumbnails for images and rasters:

```bash
ddb thumbs *.JPG -o thumbs/
ddb thumbs orthophoto.tif -o thumb.jpg -s 1024
```

### Re-processing indexed files after an upgrade

When you upgrade DroneDB to a version that extracts additional metadata or supports new file types, existing indexed files are not automatically re-processed. Use `rescan` to update all metadata in place without re-adding files:

```bash
cd my-dataset/
ddb rescan
```

Re-process only specific entry types (e.g., images and point clouds):

```bash
ddb rescan --type image,pointcloud
```

Continue processing even if individual files fail:

```bash
ddb rescan --continue-on-error
```

### Generating contour lines from a DEM

`ddb contour` generates GeoJSON contour lines from any single-band elevation raster (DEM, DSM, DTM):

```bash
ddb contour dem.tif -o contours.geojson
```

Common options:

| Option | Description |
|--------|-------------|
| `-i, --interval` | Contour interval in raster units (default: auto) |
| `-n, --count` | Number of contour levels (alternative to `--interval`) |
| `-b, --base` | Base offset for the first contour level |
| `--min` | Clip values below this threshold |
| `--max` | Clip values above this threshold |
| `-t, --tolerance` | Simplification tolerance (0 = no simplification) |
| `--band` | Band index to use (default: 1) |

```bash
# Every 10 m, simplified to 1 m tolerance
ddb contour dem.tif -o contours.geojson -i 10 -t 1

# 20 evenly spaced levels between min and max
ddb contour dem.tif -o contours.geojson -n 20
```
